# i18n — gyutron.com

Locales: **en** (default, `/`), **de** (`/de/`), **ja** (`/ja/`). There is **no Chinese locale** on
this site.

## How it works

- Dictionaries: `astro/src/i18n/{en,de,ja}.json` — flat key→string maps, **807 keys** each.
  Keys are mostly opaque numeric (`main.001` … `main.NNN`) plus namespaced ones (`seo.*`, `fnav.*`,
  `home.*`, `hero.*`). (The dicts also still contain some `shop.*` keys inherited from the legacy
  pipeline — ignore them, they belong to the storefront.)
- Lookup: `t(locale, key)` in `src/i18n/index.ts`. **Throws on a missing key** so the build fails
  loudly instead of silently shipping English.
- URLs: `localizeUrl(locale, path)` → `/path` (en), `/de/path`, `/ja/path`.
- `<head>`: `Layout.astro` emits per-locale `hreflang` alternates + `x-default` + `og:locale`.
- Product specs/tags are localized at **render time** by `src/lib/productI18n.ts` (maps ~85 spec
  labels/values/tags en→de/ja); universal notation (IP68, GigE, Wi-Fi 6E, mAh…) stays English.

`src/config/i18n.ts` re-exports the locale list/labels from `src/i18n` so there is a single source.

## Add or edit a string

1. Edit the value in **all three** dicts (`en.json`, `de.json`, `ja.json`) for the same key.
2. New visible text → add a new key to all three. A missing key breaks the build (intended gate).
3. After editing `products.*.js`, run `python tools/i18n-audit.py` (exit 0 = clean).

## Add a language (future)

1. Add the code to `i18n.locales` in `astro.config.mjs` and to `locales` in `src/i18n/index.ts`
   (+ `ogLocale`/`htmlLang`/`localeLabel`). 2. Add `src/i18n/<code>.json` (full translation).
   3. Add `src/data/products.<code>.js`. 4. Add `pages/<code>/…` wrappers. 5. Build.
See `docs/ADD-A-LANGUAGE.md` for the legacy notes.

## Gotchas

- **UTF-8**: any Python touching de/ja text must run with `PYTHONUTF8=1 PYTHONIOENCODING=utf-8`
  (the Windows console is GBK and will write `?` mojibake otherwise).
- **Never translate a product model name** (GY-A90, GY-CR720, …) — names are identical across locales.
- Legacy `locales/i18n/{de,ja}.json` is dead (only used by the retired generator) — edit the **astro**
  dicts, not those.

## Residual-English status (updated 2026-06-07, Phase 6)
- `npm run verify:i18n` (heuristic scan of all de/ja built pages) currently reports **0 suspects**.
- Most de/ja prose is localized (nav, footer, support, FAQ, legal, product `type`/`summary`). The contact
  form's status messages are per-locale via `ContactSales.astro`'s `STATUS` map.
- **Correction to the earlier "0 residual English" claim:** that scan checked only *rendered* pages with
  a small marker list and **missed product-category DATA fields that bypass the i18n dicts** — they are
  plain strings in `products.{de,ja}.js` rendered directly by `ProductPage.astro`. Phase 6 found and fixed
  that set:
  - **Rendered** (changed `public/ja/calibration-tools.html`): `calibration-tools` `panelText`
    (was "…gauge block, and light-reference tools").
  - **Not rendered** (data hygiene only, deploy-neutral): `panelMetric` "9 models" + three `sectionIntro`
    paragraphs (sensing / machine-vision / inspection-instruments) in **both** `ja` and `de`.
    `panelMetric` and `sectionIntro` are not read by any component today.
- Intentionally English (do NOT "fix"): brand `GYUTRON`, product model names (`GY-*`), universal tech
  notation (IP68, Wi-Fi 6E, Android 14, GigE, mAh, RFID/NFC/PLC…), German loanwords (Support, Hardware,
  Online, Downloads, Workflow).
